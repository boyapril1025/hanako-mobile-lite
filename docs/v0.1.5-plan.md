## 一、沉浸式状态栏 + 导航栏

### 现状

- 主题 `Theme.AppCompat.Light.NoActionBar`，状态栏白色、导航栏灰色
- `setupSystemBarInsets()` 给 rootContainer 加 padding，WebView 不延伸到系统栏下方
- 视觉上顶部一条白、底部一条灰，像穿了一件不合身的衣服

### 方案

**Android 侧：**
1. 启用 Edge-to-Edge 显示
   - 调用 `WindowCompat.setDecorFitsSystemWindows(window, false)` 让内容延伸到系统栏下方
   - API 35+ 设备可用 `enableEdgeToEdge()`，但向下兼容用 `setDecorFitsSystemWindows` 更稳
2. 修改主题
   - `statusBarColor` → `@android:color/transparent`
   - `navigationBarColor` → `@android:color/transparent`
   - `windowLightStatusBar` → `true`（浅色背景 + 深色图标）
   - `windowLightNavigationBar` → `true`
3. 移除 `setupSystemBarInsets()` 的 padding 逻辑
4. 保留 inset 监听，但改为只记录 inset 值，不设 padding

**WebView 侧（CSS 注入）：**
1. 页面加载后注入 CSS，让页面内容避开状态栏和底部手势区：
   ```css
   body, #app, .app-container {
     padding-top: env(safe-area-inset-top, 0px);
     padding-bottom: env(safe-area-inset-bottom, 0px);
   }
   ```
2. Hanako mobile web 如果已有 safe-area 处理则叠加确认，避免双倍间距

### 风险

- 如果 Hanako mobile web 自身已有 padding/margin，可能出现双重间距。需要真机调试确认，必要时用 JS 检测并覆盖。

## 二、聊天选择后侧边栏保持展开

### 现状

- 选中一个对话后，mobile web UI 的会话列表（侧边栏）自动收起，只显示聊天内容
- 在桌面端合理，但在手机端打断了"快速浏览多条对话"的流程

### 方案

**JS 注入（页面加载后执行）：**

1. 观察 DOM 结构，定位会话列表容器和聊天内容容器
2. 拦截会话选择事件，阻止默认的侧边栏收起行为
3. 关键策略：
   - 找到切换侧边栏可见性的触发逻辑（可能是某个 class 切换或 display 属性变化）
   - 在会话切换流程中，强制保持侧边栏的展开状态
   - 用 MutationObserver 监听 DOM 变化，如果侧边栏被意外隐藏则立即恢复

4. 实现思路：
   ```javascript
   // 伪代码 - 需要根据实际 DOM 结构调整
   // 找到侧边栏元素，监听其 class/style 变化
   // 当检测到"收起"状态时，如果当前是"选中对话"状态，则恢复展开
   ```

### 风险

- 依赖 mobile web UI 的 DOM 结构，如果 UI 更新可能失效
- 需要真机调试确认 DOM 结构后才能写出精确的选择器
- 如果侧边栏收起是通过路由变化（而非纯 CSS），可能需要更复杂的拦截

### 实施前提

**需要先在手机浏览器中检查 Hanako mobile web 的 DOM 结构**，确认：
- 会话列表容器的选择器
- 聊天内容容器的选择器
- 侧边栏收起/展开的触发机制（class 切换 / style 变化 / 路由变化）

## 三、侧滑返回 → 回到聊天列表

### 现状

```kotlin
// 当前逻辑：WebView 有历史就 goBack()，没有就退出 App
if (webView.canGoBack()) {
    webView.goBack()
} else {
    isEnabled = false
    onBackPressedDispatcher.onBackPressed() // 退出
}
```

- 从聊天列表进入某个对话后，侧滑返回
- 如果 WebView 没有历史记录（页面是一次性加载的 SPA），直接退出 App
- 用户预期：返回到聊天列表，而不是退出

### 方案

**JS + Kotlin 协同：**

1. JS 侧：注入一个"应用级返回"处理函数
   - 监听自定义事件 `hana-shell-back-press`
   - 收到事件后，尝试执行"回到聊天列表"操作：
     a. 如果当前在聊天详情页，触发返回到列表的 UI 操作
     b. 如果已经在列表页，标记"已无可返回"
   - 返回结果（是否已处理）给 Android 侧

2. Kotlin 侧：修改 `handleOnBackPressed()`
   ```
   优先级从高到低：
   1. 让 JS 层处理"回到聊天列表"
   2. 如果 JS 层报告已在顶层，再检查 WebView 历史
   3. 都不行，才退出 App
   ```

3. 具体交互：
   - 第一次返回 → JS 尝试回到列表
   - 如果已在列表 → 再次返回 → 退出 App
   - 需要在 JS 侧维护一个状态标志 `__hanaAtTopLevel`

### 风险

- "回到聊天列表"的实现依赖 mobile web UI 的导航逻辑
- 需要与"侧边栏保持展开"方案协同考虑
- 如果 mobile web 的返回逻辑是通过 URL hash 或路由变化，需要额外处理

## 四、下拉刷新动画 + 手感优化

### 现状

```javascript
const refresh = () => {
    window.location.reload(); // 硬刷新，白屏
};
```

- 上拉到阈值直接 `location.reload()`
- 没有视觉反馈，页面白屏闪烁
- 手感生硬，不像微信那种"拉下来有个东西在转"的流畅感

### 方案

**JS 注入重构：**

1. **下拉指示器（Pull Indicator）：**
   - 创建一个固定定位的 DOM 元素，放在页面顶部
   - 包含一个 CSS 旋转动画的圆环图标
   - 默认隐藏，下拉时渐显、上移

2. **手势跟踪改进：**
   - 监听 `touchstart` / `touchmove` / `touchend`
   - 根据下拉距离实时更新指示器位置和透明度
   - 下拉距离 < 阈值：指示器跟随手指，透明度递增
   - 下拉距离 ≥ 阈值：指示器到达固定位置，触发"准备刷新"状态

3. **触发刷新：**
   - 松手且超过阈值 → 指示器进入旋转动画
   - 0.3s 后执行 `location.reload()`
   - 刷新完成后指示器淡出

4. **动画细节：**
   - 指示器位移：CSS `transform: translateY()`，`transition: 0.2s ease-out`
   - 透明度：从 0 到 1 的线性过渡
   - 旋转：`@keyframes spin { 0% { transform: rotate(0deg); } 100% { transform: rotate(360deg); } }`
   - 回弹：松手后如果未达阈值，指示器弹回原位，`transition: 0.3s cubic-bezier(0.25, 0.46, 0.45, 0.94)`

5. **优化手势检测：**
   - 当前逻辑要求"从底部上拉"，改为"在聊天消息列表顶部下拉"
   - 检测聊天滚动容器是否在顶部（`scrollTop <= 0`），是则允许触发
   - 更符合用户直觉

## 五、App 图标

### 现状

- 当前使用 Android 默认图标（系统紫色机器人）
- 没有自定义 launcher icon

### 方案

1. **设计阶段：**
   - 与阿哲确认图标风格方向（见下方备选）
   - 生成 1024x1024 源图

2. **备选风格：**
   - **A. 温润玉石** — 青白色调，圆润形体，底部一簇淡火，暗合"铸剑"与"温润"的底色
   - **B. 极简对话** — 纯色背景上一个精致的对话气泡，气泡内隐约一个"花"字或花形，干净利落
   - **C. 炉火微光** — 深色底，中心一簇小而亮的火苗，像铸剑炉里最后那一把火

3. **适配阶段：**
   - 生成 Android 全套尺寸：mdpi(48) / hdpi(72) / xhdpi(96) / xxhdpi(144) / xxxhdpi(192)
   - 放入 `res/mipmap-*` 目录
   - API 26+ Adaptive Icon（前景 + 背景图层）
   - 更新 `AndroidManifest.xml` 的 `android:icon` 和 `android:roundIcon`

### 依赖

- 需要阿哲选定风格方向后才能生成
- 图标设计可与其他改动并行，不阻塞开发

## 依赖关系

```
1. 沉浸式状态栏 ──┐
                   ├── 可并行开发
2. 下拉刷新动画 ──┘

3. 侧边栏保持 ────┐
                   ├── 需要先调研 DOM 结构，再协同实现
4. 侧滑返回 ──────┘
```

**建议执行顺序：**
1. 先做沉浸式（纯 Android 侧，可控）
2. 再做下拉刷新动画（纯 JS 注入，可控）
3. 然后调研 mobile web DOM 结构
4. 最后做侧边栏 + 侧滑返回（需要 DOM 调研结果）

## 验收标准

- [ ] 状态栏透明，内容延伸到状态栏下方，标题文字不被状态栏图标遮挡
- [ ] 底部导航栏透明，内容延伸到手势区上方，底部可点击元素不被手势条遮挡
- [ ] 选中对话后，侧边栏保持展开（或至少不强制收起）
- [ ] 侧滑返回回到聊天列表，不退出 App
- [ ] 下拉刷新有指示器动画，不白屏
- [ ] A/B/C 回归仍然通过（不影响 V0.1 已有功能）
- [ ] App 图标显示正常，不同密度设备不模糊
- [ ] 编译通过，debug APK 可安装运行
